# Companion: KuaiRand Schema → Notation Map + Concrete Preprocessing (Click Outcome)

This is a companion to `kuairand_click_fatigue_habit_semiparametric.md`.

It does **two practical things**:

1. Maps each theoretical symbol \\((A_{it}, W_{it}, S_{it}, \\ldots)\\) to **exact KuaiRand column names** and files.
2. Gives a **concrete preprocessing recipe** (sessions, continuation, stocks, embeddings `a_j`, and randomized bucket usage).

Sources: KuaiRand official README/data description. (See dataset website / GitHub README.)  

---

## 1) Which files to use (KuaiRand-1K / 27K / Pure)

KuaiRand ships in three versions, each with the same *schema* but different scale. The typical workflow:

- **KuaiRand-1K**: fastest for prototyping the full pipeline (sequential + OPE/RL).
- **KuaiRand-27K**: for final experiments (computationally heavy).
- **KuaiRand-Pure**: small item universe, but sequential logs are “incomplete” for some tasks; use only if you don't need full long-horizon sequential dynamics.

### 1.1 Logs (main objects for actions and outcomes)

The README lists three log files (per version):

- `log_standard_4_08_to_4_21_*.csv`  (standard recommendation, earlier two weeks; **use as history**)
- `log_standard_4_22_to_5_08_*.csv`  (standard recommendation, random-exposure stage)
- `log_random_4_22_to_5_08_*.csv`    (interactions resulting from random intervention)

These correspond to the time periods described in the KuaiRand paper and README.

### 1.2 Side information

Per version:

- `user_features_*.csv`
- `video_features_basic_*.csv`
- `video_features_statistic_*.csv`  (large)

Supplementary (extra semantic content, separate download):

- `kuairand_video_captions.csv`
- `kuairand_video_categories.csv`

These can be joined to the main data via a re-indexed video id key (see §4.4).

## 2) Exact log schema (columns in `log_xxx.csv`)

Each row is **one impression** (a recommended/exposed video) with multiple feedback signals.

> NOTE: The README contains an obvious typo “user_id The ID of the video”; interpret `user_id` as user id.

### 2.1 Action, time, environment (what we treat as “state” inputs)

| Theoretical | Meaning | KuaiRand column | Type |
|---|---|---:|---|
| \\(i\\) | user index | `user_id` | int64 |
| \\(A_{it}\\) | shown item/video | `video_id` | int64 |
| \\(t\\) | time index (per user) | derived: sort by `time_ms` | int |
| timestamp | absolute time | `time_ms` | int64 |
| date | day | `date` | int64 |
| within-day time | HHSS | `hourmin` | int64 |
| scenario/policy context | which tab/page | `tab` (0–14) | int64 |
| randomized bucket | random exposed? | `is_rand` | int64 |

**Key interpretation**:
- `is_rand=1` means this impression’s video was generated by the random intervention (randomly exposed).  
- `log_random_4_22_to_5_08_*.csv` contains interactions **resulting from random intervention**; `is_rand` is also present there.

### 2.2 Binary click outcome \\(W_{it}\\)

You requested: **let the outcome be click**.

KuaiRand provides:
- `is_click`: binary feedback; in a two-column UI it indicates “click”, and in a single-column UI it is defined as “valid_play” using `play_time_ms` and `duration_ms`.

So for our paper’s *binary engagement outcome*:
\\[
W_{it} := \\texttt{is\_click}.
\\]

### 2.3 Other feedback signals (optional covariates or multi-task extensions)

Binary:
- `is_like`, `is_follow`, `is_comment`, `is_forward`, `is_hate`, `is_profile_enter`, `long_view`

Continuous:
- `play_time_ms`, `duration_ms`, `profile_stay_time`, `comment_stay_time`

## 3) Mapping model notation to KuaiRand fields

### 3.1 Core variables

| Model symbol | Definition in our model | KuaiRand |
|---|---|---|
| \\(A_{it}\\) | action / item shown | `video_id` |
| \\(W_{it}\\) | click/engagement | `is_click` |
| \\(X_{it}\\) | context/time covariates | `tab`, `date`, `hourmin`, derived session position |
| \\(X_{A_{it}}\\) | item baseline features | from `video_features_basic_*` and/or `video_features_statistic_*` |
| \\(\\mathbf{1}\\{\\text{rand}\\}\\) | randomized exposure flag | `is_rand` (and membership in `log_random...`) |

### 3.2 What is “state” \\(S_{it}\\) in practice?

In our semiparametric formulation:
\\[
S_{it} = (h_{it}, F_{it}, G_{it}, X_{it})
\\]
where:

- `h_{it}` = a learned user-history representation (DIEN/BERT4Rec encoder output).
- `F_{it}` = fatigue stock (constructed from history + item semantic embedding `a_j`).
- `G_{it}` = habit stock (constructed from past clicks).
- `X_{it}` = observed context (`tab`, time-of-day, session position).

## 4) Concrete preprocessing recipe (this is the part you’ll actually code)

### 4.1 Load and concatenate logs (recommended)

For sequential modeling during the random-exposure stage:

1. Load `log_standard_4_08_to_4_21_*.csv`  (**history window**)
2. Load `log_standard_4_22_to_5_08_*.csv`  (**standard exposures**)
3. Load `log_random_4_22_to_5_08_*.csv`    (**random interventions**)

Concatenate them (stack rows), keep a flag for which file each row came from.

### 4.2 Sort and create per-user time index \\(t\\)

For each user:
- Sort by `time_ms` ascending.
- Create `t = 1..T_i` as row order.

This defines the dynamic sequence.

### 4.3 Sessions and continuation indicator \\(C_{it}\\)

KuaiRand logs do **not** directly provide a “stop watching” label. You need a standard proxy.

**Sessionization**:
- For each user, compute \\(\\Delta_{it} = time\_ms_{i,t+1} - time\_ms_{i,t}\\).
- Start a new session when \\(\\Delta_{it} > \\Delta\\_\\text{gap}\\) (common: 30 minutes).

Define:
- `session_id` per user
- within-session position `pos_in_sess`
- session length `L_sess`

**Continuation**:
\\[
C_{it} := \\mathbf{1}\\{\\text{next impression exists in same session}\\}.
\\]
So the last impression in each session has `C_{it}=0`.

> Why this is the right object: our structural mechanism needs “fatigue → quitting” (stopping hazard). Session end is the cleanest proxy without additional labels.

### 4.4 Join item semantic information for building `a_j`

You have three reasonable sources for item semantics:

1. `video_features_basic_*.csv`: has `tag` (string like `"12,65"`) and some item metadata.
2. `kuairand_video_categories.csv`: hierarchical categories + probabilities.
3. `kuairand_video_captions.csv`: raw caption text + cover text.

**Join keys**
- Logs use `video_id`.
- Supplementary files use `final_video_id` and claim consistency with KuaiRand.

In most installations, `video_id` is the re-indexed id. In practice you should:
- Try joining `log.video_id == captions.final_video_id` and check match rate.
- If mismatch occurs, look for a provided mapping file in the Zenodo bundle, or infer mapping from overlaps.

### 4.5 Construct semantic embedding \\(a_j\\) (fatigue attributes)

You want `a_j` to represent *the dimension along which homogeneous exposure creates fatigue*.

Practical constructions:

**Option A (fast, strong baseline): multi-hot over `tag`**
- Parse `tag` into a set of integer tag IDs.
- Create a multi-hot vector over tag universe, then L2 normalize.

**Option B (hierarchical categories):**
- Use `first_level_category_id` ... `fourth_level_category_id` plus their probabilities.
- Build a vector that includes either:
  - the most probable leaf-level category id, or
  - a probability-weighted multi-level vector.

**Option C (text semantics):**
- Embed captions with a text encoder (e.g., sentence transformer), reduce to dimension `d` with PCA, normalize.

**Recommended hybrid**:
- Concatenate (tags multi-hot) + (category prob vector) + (caption embedding) then project to `d`.

> Identifiability rule: keep `a_j` reserved for fatigue. If your baseline click model directly consumes exactly the same high-dimensional `a_j`, it can absorb the fatigue effect and you will estimate ~0. Use different “views” of content for baseline vs fatigue (or regularize).

## 5) Mapping the stocks \\(F_{it}\\) and \\(G_{it}\\) to KuaiRand

### 5.1 Habit stock \\(G_{it}\\)

With click outcome:
\\[
G_{i,t+1} = (1-\\delta_G) G_{it} + W_{it}\\,\\mathbf{1}\\{C_{it}=1\\},\\quad G_{i1}=0.
\\]
Use `W_{it} = is_click`.

### 5.2 Fatigue stock \\(F_{it}\\)

Let `a_{A_{it}}` be the semantic embedding for `video_id`:

\\[
F_{i,t+1} = (1-\\delta_F) F_{it} + a_{A_{it}}\\,\\mathbf{1}\\{C_{it}=1\\},\\quad F_{i1}=0.
\\]

### 5.3 Fatigue index used in the click equation

\\[
T_{it} = \\langle F_{it}, a_{A_{it}}\\rangle.
\\]

This is the “homogeneity score”: large when the user is repeatedly served similar content.

## 6) How to exploit KuaiRand’s random exposure (`is_rand`) in estimation

### 6.1 Why this matters

In standard logs, item choice is confounded by the platform’s recommender. KuaiRand injects random exposures so that for those impressions, selection is **closer to exogenous** given state.

### 6.2 Two practical strategies (use both)

**Strategy 1 (clean identification of fatigue effect):**
- Fit the fatigue sensitivity \\(\\lambda(S)\\) primarily on impressions with `is_rand=1`.
- Fit the baseline representation `h_{it}` / nuisance `g(S,X_A)` using all data.

**Strategy 2 (DR OPE with support):**
- Restrict policy learning and off-policy evaluation to:
  - candidate sets that appear in logs, or
  - actions with nontrivial support in random exposure bucket.

### 6.3 Propensities (what you can and cannot do)

KuaiRand does **not** provide explicit propensities \\(\\pi_0(a|S)\\) for standard recommendations.

For OPE, you have options:
- Estimate a propensity model on logs (multinomial / softmax over a candidate set proxy).
- Use randomized bucket where propensity is close to uniform over the random pool (requires knowing/constructing the pool).

The README reports a “random policy” item pool size (e.g., ~7.5k items for some versions); use that as your uniform support set **only if** it matches your installed version’s pool.

## 8) Minimal, implementation-ready pseudocode

```python
import pandas as pd
import numpy as np

GAP_MS = 30 * 60 * 1000  # 30 minutes

# 1) Load logs
log_hist = pd.read_csv("log_standard_4_08_to_4_21_1k.csv")
log_std  = pd.read_csv("log_standard_4_22_to_5_08_1k.csv")
log_rand = pd.read_csv("log_random_4_22_to_5_08_1k.csv")

log_hist["source"] = "std_pre"
log_std["source"]  = "std"
log_rand["source"] = "rand"

df = pd.concat([log_hist, log_std, log_rand], ignore_index=True)

# 2) Sort per user
df = df.sort_values(["user_id", "time_ms"]).reset_index(drop=True)

# 3) Sessionize
df["time_next"] = df.groupby("user_id")["time_ms"].shift(-1)
df["gap_next"] = df["time_next"] - df["time_ms"]
df["new_sess_next"] = (df["gap_next"] > GAP_MS) | df["gap_next"].isna()

# continuation C_it = 1 if next exists in same session
df["C"] = (~df["new_sess_next"]).astype(int)

# define session_id by cum-summing "new session starts" flags
df["new_sess_start"] = (
    (df.groupby("user_id")["time_ms"].diff().fillna(GAP_MS+1) > GAP_MS)
).astype(int)
df["session_id"] = df.groupby("user_id")["new_sess_start"].cumsum()
df["pos_in_sess"] = df.groupby(["user_id","session_id"]).cumcount() + 1

# 4) Outcome W = is_click
```

## 9) Quick “gotchas” specific to KuaiRand

1. **Meaning of `is_click` depends on UI**. If you want “true click”, you may prefer using `is_profile_enter` or `is_like`, or define your own click proxy; but since you requested binary clicks, we set \\(W=is\\_click\\).
2. **Heavy tails in `play_time_ms` and `duration_ms`**: if you switch to watch-time outcomes later, you’ll need trimming / log transform.
3. **`video_features_statistic_*` is huge**: for prototyping, start with `video_features_basic_*` + tags/captions/categories.
4. **Random exposure is rare** (~0.37% in the description): you’ll want careful variance control and/or combine with standard logs for representation learning.
